abstraction layer for multi-GPU backend support in the Blender Remote GPU
renderer. Users can now seamlessly switch between OptiX, CUDA, HIP, and

Key achievement: Automatic backend detection + runtime selection + graceful
fallback + unified Blender UI for backend switching.

================================================================================
BACKEND DETECTION LOGIC
================================================================================

OptiX Detection:
  ✓ nvidia-smi --query-gpu=driver_version  (NVIDIA driver)
  ✓ nvcc --version                         (CUDA toolkit)
  → Available if both present, RTX GPU detected

CUDA Detection:
  ✓ nvcc --version                         (CUDA toolkit)
  ✓ nvidia-smi                             (GPU list)
  → Available if both present, fallback from OptiX

HIP Detection:
  ✓ hipcc --version                        (ROCm compiler)
  ✓ rocm-smi                               (GPU list)
  → Available if both present, AMD GPUs only

Vulkan Detection:
  ✓ vulkaninfo --summary                   (Vulkan SDK)
  → Available if Vulkan drivers installed

Priority Order (for default):
  1. OptiX (fastest on RTX Ampere/Ada/Blackwell)
  2. CUDA (fallback for older NVIDIA)
  3. HIP (AMD ROCm)
  4. Vulkan (multi-vendor, experimental)

================================================================================
PROTOCOL INTEGRATION
================================================================================

New Message: RENDER_DEVICE_SELECT (0x12)
  Direction: Client → Server
  Payload: {"backend": "OPTIX" | "CUDA" | "HIP" | "VULKAN", "device_idx": 0}
  Usage: User clicks "Switch" button in Blender addon UI

Extended: STATUS Message (0x90)
  Direction: Server → Client (sent on connection + after device select)
  New fields:
    - available_backends: ["OPTIX", "CUDA", ...]
    - current_backend: "OPTIX"
    - backend_devices: [{"name": "RTX 5090", "vram_gb": 24.0, ...}, ...]

Data Flow:
  1. Client connects
  2. Server detects available backends
  3. Server sends STATUS with backend info
  4. Client renders UI with backend list
  5. User clicks backend button
  6. Client sends RENDER_DEVICE_SELECT
  7. Server validates + selects backend
  8. Server sends updated STATUS
  9. Client UI updates
  10. Next render uses selected backend

================================================================================
RENDERING INTEGRATION
================================================================================

When SubprocessRenderer.render_final() is called:
  1. _build_render_script() queries gpu_backend_mgr.get_current_device_arg()
     → "optix" (or "cuda", "hip", "vulkan")

  2. Generated Blender Python script contains:
     ```python
     backend_priority = ["optix", "OPTIX", "CUDA"]  # selected first
     for device_type in backend_priority:
         try:
             cprefs.compute_device_type = device_type
             cprefs.get_devices()
             # Success → break and render with this backend
         except:
             continue  # Try next in priority list
     ```

  3. Blender renders using selected backend (or next available if fails)

  4. Server logs: "Rendering with backend device: optix"

Benefits:
  ✓ User-selected backend used preferentially
  ✓ Graceful fallback if selected backend fails
  ✓ No render failure if GPU removed/unavailable
  ✓ Full backward compatibility (existing behavior preserved)

================================================================================
ERROR HANDLING
================================================================================

Scenario 1: User selects unavailable backend
  Client: RENDER_DEVICE_SELECT {"backend": "HIP"}
  Server: Backend not in available list
  Server: Calls fallback_to_available() → switches to CUDA
  Server: Sends STATUS with current_backend="CUDA"
  Result: Render uses CUDA instead, user informed via UI

Scenario 2: Selected backend fails at render time
  Blender script tries: ["cuda", "OPTIX", "CUDA"]
  CUDA initialization fails
  Script tries OPTIX → success
  Result: Render completes on fallback backend

Scenario 3: GPU driver missing
  Backend.validate() returns (False, "CUDA toolkit (nvcc) not found")
  Backend not added to available_backends
  User never sees unavailable backend in UI
  Result: Only working backends offered

Scenario 4: Multiple backends available, default selected
  Detection finds: OptiX, CUDA, HIP
  Default: OptiX (first in priority)
  User can switch to CUDA or HIP from UI
  Result: Maximum flexibility
