PROTOCOL SPECIFICATIONS
================================================================================

PASS_DATA (0x24) - Individual Pass Delivery
  {
    "type": 0x24,
    "pass_name": str,              # "Beauty", "Normal", "Depth", etc.
    "channel_count": int,          # 1, 3, or 4
    "width": int,                  # Pixels
    "height": int,                 # Pixels
    "data_format": str,            # "RAW_FLOAT", "RAW_INT", "JPEG", "PNG"
    "tier": int,                   # 1-4 (PassTier.value)
    "timestamp": float,            # Seconds since render start
    "binary_len": int,             # Binary payload size
  }
  [binary: compressed pass data bytes]

PASS_COMPLETE (0x25) - All Passes Delivered
  {
    "type": 0x25,
    "total_passes": int,           # Total passes sent
    "total_bandwidth_mb": float,   # Cumulative bandwidth
  }

================================================================================
TIER SCHEDULE
================================================================================

Tier 1 - Beauty Preview (0-2s)
  Passes: Beauty
  Format: JPEG (quality 85%)
  Size: ~10 MB
  Purpose: Instant viewport preview
  Trigger: Compositor shows preview

Tier 2 - Essential (2-10s)
  Passes: Normal, Depth, Diffuse Color
  Format: PNG (40-60% compression)
  Size: ~35 MB
  Purpose: Color correction, depth effects
  Trigger: Compositor becomes fully active

Tier 3 - Full Passes (10-30s)
  Passes: Glossy, Specular, Shadow, AO, Emission
  Format: PNG compression
  Size: ~50 MB
  Purpose: Material breakdown
  Trigger: Full material controls available

Tier 4 - Cryptomatte (30-40s)
  Passes: Cryptomatte ID + metadata JSON
  Format: PNG + JSON
  Size: ~5 MB
  Purpose: Keying and selection tools
  Trigger: Advanced keying unlocked

================================================================================
DESIGN HIGHLIGHTS
================================================================================

1. Async Architecture
   - Beauty sent immediately (blocking)
   - Remaining passes streamed asynchronously
   - Non-blocking to main render handler

2. Tier-Based Scheduling
   - Predictable delivery timeline
   - Respects network latency with time windows
   - Each tier optimized for specific purpose

3. Live Compositor Integration
   - Compositor re-evaluates only affected nodes
   - No full graph re-evaluation needed
   - No additional render passes required

4. Compression Optimization
   - JPEG for beauty (10x compression)
   - PNG for float data (40-60% compression)
   - JSON for metadata (minimal overhead)

5. Modular Design
   - Each component independently testable
   - Clear separation of concerns
   - Easy to extend with new passes

6. EXR Agnostic
   - Placeholder for OpenEXR/imageio/custom parsers
   - Flexible pass extraction

7. Error Resilience
   - Graceful fallbacks
   - Detailed error logging
   - Client can skip problematic passes

8. Backward Compatible
   - Existing FRAME_FINAL pipeline still works
   - Progressive passes are optional enhancement
   - Can be disabled without code changes
